About

What is Startup Staff?

Startup Staff — Founder Edition is an AI-powered boardroom for founders and builders. Instead of querying a single AI model, you get a cross-functional team of seven specialist personas that collaborate on every question you ask.

Every query flows through a three-phase agentic pipeline: your Chief of Staff decides which specialists are relevant, each specialist responds independently using their assigned AI model and the full recent conversation history, and the Chief of Staff synthesises their perspectives — weighting the most relevant expert's view — into a single unified recommendation.

Startup Staff is a pure client-side single-page application — one HTML file (~6,600 lines) with no build step and no backend beyond Supabase for auth and settings sync. You bring your own API keys across 12 different providers; we never see your prompts.

Think of it as a fractional C-suite — Engineering, Finance, Marketing, Legal, Architecture, and HR — available any time, for any question. Each member can run on a different AI provider and model.

How It Works

Every message flows through a three-phase agentic pipeline. Specialists receive the last 10 turns of conversation history — they know what's been discussed.

Phase 1 — Routing
The Chief of Staff reads your message and outputs [CONSULT: Name] tags for relevant specialists. She's selective: tax → Finance only; contract → Legal; landing page → Marketing + Architecture + Engineering. Say "full team discussion" or "everyone's input" to skip Phase 1 and consult all 6 simultaneously.
Phase 2 — Specialist Responses
Each consulted specialist responds independently and streams live into chat. They see the full recent conversation. Domain weighting is applied: Finance leads on pricing, Legal on contracts, Marketing on GTM. Engineering gets 8,192 tokens when generating documents.
Phase 3 — Synthesis
The Chief of Staff synthesises all specialist responses into a unified recommendation, weighted toward the lead-voice domain expert. For document requests, Engineering's [DOCUMENT: HTML] block is extracted from the specialist responses and a download card is injected into chat.
A single query can trigger up to 9 API calls: 1 routing + up to 6 specialists + 1 synthesis + 1 session memory update. Each call is billed against the assigned provider's API key.
Getting Started

Quick Start

From sign-in to your first boardroom response in under five minutes:

01
Sign in with LinkedIn

Authentication via LinkedIn OAuth through Supabase. No password, no email confirmation. Your account is created automatically on first sign-in.

02
Complete onboarding

Your Chief of Staff will ask for your name, startup description, and HQ country. This context is injected into every AI prompt. Update anytime in Settings → Startup & Business.

03
Add at least one AI provider

Open Settings → AI Providers, enter your API key, click Test (probes online/offline status and training cutoff), then use Set All to assign one model to the whole team or configure individually.

04
Ask your first question

Type any strategic question — "Should we raise a seed round now?", "Generate a landing page for our product", "Review our SAFE agreement terms." The boardroom handles routing automatically.

05
Enrich with Memory and Knowledge Base

Give individual specialists reference documents in Settings → Team Configuration → Staff KB. Add permanent context facts in the Memory panel for cross-session retention.

Asking Questions

No special commands required. Type naturally. A few tips:

💬
Be specific about your situation

The more context, the better. Responses are far more useful when your company description and HQ country are set in Settings.

👥
Request full-team input

Say "I need everyone's input", "team discussion", or "all perspectives" to skip Phase 1 routing and consult all 6 specialists in parallel.

🔄
Follow up naturally

Each specialist sees the last 10 turns of conversation. Push back, ask for clarification, or drill deeper — the team retains context throughout the session.

Using the App

Hint System

Hints are keywords that trigger specific system actions. They can be combined freely in one message — the parser detects them independently.

generate a html [topic]
Generates a branded HTML page. Engineering builds using content from the domain expert + design from Architecture.
generate a pdf [topic]
Same as HTML but print-ready with @media print CSS — open and File → Print → Save as PDF.
generate a [contract / report / JD / budget]
Auto-detects domain: contract/terms → Legal · job/hire → HR · budget/forecast → Finance · pitch/GTM → Marketing.
email it to x@example.com
Sends generated document to the specified address via Resend (requires Edge Function — see roadmap).
subject [text] · body [text]
Sets subject and body for email sends. Combine with email hint in one message.
send a telegram: [message]
Sends a Telegram message instantly via your configured bot. Works directly from the browser.
send that on telegram
Sends the most recently generated document as a Telegram notification (expires after 30 min).
send me the last summary on telegram
Sends the Chief of Staff's last synthesis to Telegram (expires after 1 hour).
Hints combine: "generate a landing page for our product and email it to ceo@company.com" — generates the HTML, injects a download card, and emails it, all in one shot.
Using the App

Document Generation

Document requests flow through the normal Phase 1→2→3 pipeline with a three-role collaboration model — content, design, and build are separated.

Domain Expert — Content Brief
The relevant specialist provides the content brief — key messages, required sections, domain-specific requirements, and tone. Legal for contracts/T&C/privacy · HR for job descriptions · Finance for financial docs · Marketing for pitches and landing pages · COS for general reports.
Architecture — Design Guidance
Plato provides layout and design guidance — visual structure, section hierarchy, typography choices, brand color usage from the Startup Staff palette, and UX considerations for the document type.
Engineering — Final Generation
Descartes generates the final branded HTML using the content brief, design guidance, and full conversation context. Output uses the Startup Staff brand (🚀 logo, Geist fonts, dark/light theme). A download card is injected into chat immediately. 8,192 token limit for document generation.

Create a bot via @BotFather

Open Telegram, search for @BotFather, send /newbot. You'll receive a token like 7123456789:AAF...

02

Start a chat with your new bot (send it "hi"), then visit https://api.telegram.org/bot<TOKEN>/getUpdates. Your chat ID is the number under chat.id.

Using the App

Online / Offline Detection

Startup Staff probes each staff member's assigned model on login and after saving assignments to detect whether it has live internet access.

🔍
Date probe

A silent background request asks each model: "What is today's exact date? Reply YYYY-MM-DD only." Staggered 300ms apart. Within 3 days of today = online. More than 3 days off = offline + cutoff recorded.

🌐
Bubble footer badges

Every chat bubble shows 🌐 Web Search (online) or 📚 Training Knowledge · cutoff YYYY-MM (offline). Online models with citations show a clickable badge opening a sources modal with URLs.

⚡
Test button also probes

The Test button in Settings → Providers uses the same date probe and shows the result immediately as Connected · gpt-4o · 🌐 Online or Connected · model · 📚 Offline · cutoff 2024-01.

Perplexity (all sonar models), xAI Grok (grok-4+), and OpenAI search models return live web citations via their APIs. These are extracted from the response and shown in the clickable sources modal.

Startup Staff has two layers of memory, both injected into every AI prompt:

Default MemoryPermanent facts — key decisions, product context, team structure, non-negotiables. Edited in the Memory panel.Supabase DB — survives reload and re-login

Chat history itself is not persisted — it lives in browser memory and resets on reload. Use the ↓ Save Chat button (beside the trash icon in the conversation header) to download the full session as a branded HTML file.
Using the App

Knowledge Base

Each specialist can have a private knowledge base injected only into their system prompt — not shared with other staff. Give Legal your contract templates, Engineering your tech spec, Finance your financial model.

  • Reference URLs — one per line. Fetched at sign-in (CORS permitting). Works best for public docs, GitHub READMEs, and your own hosted content.
  • Uploaded documents — .txt, .md, .csv, .json files. Content is read into the KB on save.
URL fetching happens directly from the browser and fails for most sites due to CORS. Errors appear as [Fetch failed]. For reliable knowledge injection, use uploaded files.

Roadmap

Future Improvements

Areas planned for improvement, roughly in order of impact:

01
Email via Supabase Edge Function Security
Resend's API blocks direct browser calls due to CORS. A thin Supabase Edge Function would proxy the call server-side. The API key is already saved and the UI is fully wired — only the Edge Function deployment is pending.
02
Persistent chat history Feature
Chat history resets on every page reload. Saving conversations to Supabase (with a sessions table) would enable history browsing, resume after closing the tab, and conversation search.
03
API key proxy / backend Security
A thin server-side proxy would make LLM calls on behalf of the user — keeping API keys entirely off the client. The single biggest security improvement possible.
04
Telegram contact list Feature
Currently sends to a single configured chat ID. A contacts list (name + chat ID pairs) would let you say "telegram Mike: the report is ready" and route to the right person automatically.
05
Token budget management Cost
No token counting or context pruning. As chat history grows, prompts become very large. A sliding window with summarisation would control costs and prevent context overflow on smaller models.
06
Multi-tab / multi-device sync Reliability
Settings changed in one tab aren't reflected in other open sessions without a reload. A Supabase Realtime subscription, or a BroadcastChannel, would keep all sessions in sync.
07
Mobile-optimised layout Accessibility
The boardroom SVG and dual-panel layout are designed for desktop. A rethought mobile layout — collapsible team panel, bottom chat bar, card-based responses — would open Startup Staff to on-the-go use.
08
Modular codebase Maintainability
The app is ~6,600 lines in a single HTML file. A minimal build step (Vite + vanilla JS) would enable code splitting and proper imports without losing the zero-backend philosophy.
Roadmap

Known Limitations

API keys in browser memory. window.providerConfigs holds all API keys in plaintext during the session. Any JS running on the page can read them. Always set usage limits at your provider's dashboard.
No rate limiting. A single query with all six specialists can trigger up to 9 API calls. Use provider-level rate limits and spending caps as a safeguard.
Context window growth. chatHistory is never pruned. Long conversations will eventually exceed model context windows. Session summarisation partially mitigates this.
Resend email requires Edge Function. Email delivery is fully configured in the UI but blocked by CORS until the Supabase Edge Function is deployed. See roadmap #01.
URL KB fetch CORS. URL fetching fails for most sites due to browser CORS policies. Use uploaded files for reliable knowledge injection.
AI responses are not professional advice. Legal, financial, and other specialist outputs are starting points — not substitutes for qualified professionals.
