What is Startup Staff?

Startup Staff — Founder Edition is an AI-powered boardroom for founders and builders. Instead of querying a single AI model, you get a cross-functional team of seven specialist personas that collaborate on every question you ask.

Every query is routed through a three-phase pipeline: your Chief of Staff decides which specialists are relevant, each specialist responds independently using their assigned AI model, and the Chief of Staff synthesises their perspectives into a single unified recommendation — complete with action items and a clear line of reasoning.

Startup Staff is a pure client-side single-page application — one HTML file with no build step and no backend beyond Supabase (for auth and settings sync) and direct API calls to the AI providers you configure. You bring your own API keys; we never see your prompts.

Think of it as hiring a fractional C-suite — Engineering, Finance, Marketing, Legal, Architecture, and HR — available any time, for any question about your startup.

How It Works

Every message you send goes through a three-phase multi-agent pipeline. Each phase is a separate API call to the AI model assigned to that staff member.

Phase 1 — Routing
Your message goes to the Chief of Staff along with your startup context and memory. The CoS decides which specialists to consult for this query and responds with [CONSULT: Name] tags. Specialists not mentioned are skipped — saving tokens and latency.
Phase 2 — Specialist Consultation
Each consulted specialist receives the original query plus their tailored system prompt (including their persona, any knowledge base content you've added, and memory). They respond independently on their assigned AI model. Calls are made sequentially. Up to 6 specialists can be consulted in a single query.
Phase 3 — Synthesis
The Chief of Staff receives all specialist responses and synthesises them into a unified recommendation — clear paragraphs, bullet action items, and a direct conclusion. This is the response you see in the chat. The CoS also generates a brief session memory entry to maintain continuity.
A single query can trigger up to 9 API calls: 1 routing + 6 specialists + 1 synthesis + 1 session summary update. Each call is billed against the API key of the assigned provider. Use the Status dots on the boardroom SVG to check which staff members are active.
Getting Started

Quick Start

From sign-in to your first boardroom response in under five minutes:

01
Sign in with LinkedIn

Authentication is via LinkedIn OAuth through Supabase. No password, no email confirmation. Your account is created automatically on first sign-in.

02
Complete onboarding

Donna (your Chief of Staff) will ask for your name, a description of your startup, and your HQ country. This context is injected into every AI prompt — it's what makes responses feel relevant to your specific situation. You can update this any time in Settings.

03
Add at least one AI provider

Open Settings → AI Providers and enter an API key for at least one provider. Use Set All to assign a single model to all staff at once, or configure each member individually. The status dots on the boardroom turn green once a staff member is configured.

04
Ask your first question

Type any strategic question into the chat — "Should we raise a seed round now or wait?", "What's our best GTM strategy for Southeast Asia?", "Review our SAFE agreement terms." The boardroom will handle the routing automatically.

05
Enrich with Memory and Knowledge Base

Use the 🧠 Memory button to add permanent context (key decisions, product facts, team structure). Use Staff KB in Settings to give individual specialists reference documents or URLs — e.g. your term sheet for the Legal Lead, your technical spec for the Engineering Lead.

Using the App

Asking Questions

There are no special commands required. Just type naturally. The Chief of Staff will route your query to the right specialists automatically based on the content. A few tips:

💬
Be specific about your situation

The more context you give, the more useful the responses. "Should we incorporate in Singapore or Delaware?" will get a better answer if your HQ country and business description are set.

📄
Request documents

Ask to "generate a report on...", "create a spreadsheet with...", or "write an HTML page for..." and the app will produce a downloadable .docx, .xlsx, or .html file attached to the response.

🎯
Direct queries to a specific specialist

You can prompt specific staff: "the Legal Lead, review this clause..." or "the Finance Lead, model our runway if we hire 3 engineers." The Chief of Staff will respect explicit routing signals.

🔄
Follow up naturally

Chat history is maintained within the session and passed to the CoS in each prompt. You can follow up, push back, or ask for more detail and the team will retain context from earlier in the conversation.

Using the App

Memory System

Startup Staff has two layers of memory, both injected into every AI prompt:

LayerWhat it isHow to manage
Default Memory Permanent facts you want every session to know — key decisions, product context, team structure, non-negotiables. Persisted to Supabase. Edit freely in the 🧠 Memory panel. Use Quick Add for fast additions.
Session Summary An auto-generated summary of the current session, updated every few turns. Helps the CoS maintain continuity within a long conversation. View in the Memory panel. Promote to Default Memory to preserve insights across sessions.
Chat history itself is not persisted — it lives in browser memory only and resets on page reload. Only Default Memory and Session Summaries you promote are saved across sessions.
Using the App

Knowledge Base

Each specialist can be given a personal knowledge base — context that is injected only into their system prompt, not shared with others. This lets you give each specialist exactly the reference material they need.

You can add two types of content per specialist:

  • Reference URLs — one per line. The app will attempt to fetch and extract the text content at sign-in (browser CORS permitting). Most sites block cross-origin requests, so this works best for public docs, GitHub READMEs, and your own hosted content.
  • Uploaded documents — .txt, .md, .csv, and .json files. Content is read into the knowledge base on save.
URL fetching happens directly from your browser and will fail for most sites due to CORS policies. Errors are embedded as [Fetch failed] notices in the KB content. For reliable knowledge injection, use uploaded files instead.

Document Exports

When the app detects a document request in your message, it instructs the Chief of Staff to embed a structured block in its synthesis response. Once the response arrives, a download card appears in the chat.

Roadmap

Future Improvements

Areas we're planning to improve, roughly in order of impact:

01
Streaming responses UX
All API calls currently use non-streaming mode. Implementing SSE streaming — especially for the Phase 3 synthesis — would dramatically reduce perceived wait time. The synthesis is the longest response and the one users wait on most.
02
Parallel specialist calls Performance
Phase 2 specialist calls run sequentially. Switching to Promise.all() would cut total latency from N × specialist_latency to max(specialist_latency) — a major win when 3+ specialists are consulted.
03
Persistent chat history Feature
Chat history resets on every page reload. Saving conversations to Supabase (with a sessions table) would enable history browsing, resume after closing the tab, and conversation search — all high-value for returning users.
04
API key proxy / backend Security
A thin server-side proxy (Supabase Edge Function or Cloudflare Worker) would make LLM calls on behalf of the user — keeping API keys entirely off the client. This is the single biggest security improvement possible.
05
CORS proxy for URL knowledge base Reliability
The Staff KB URL fetcher hits external URLs directly from the browser, which fails for most sites due to CORS policies. A server-side fetch proxy would make URL-based knowledge injection work reliably.
06
Token budget management Cost
There's no token counting or context pruning. As chat history grows, prompts become very large — eventually exceeding model context windows. A sliding window on chat history (after summarisation) would control costs and prevent overflow errors on smaller models.
07
Per-specialist typing indicators UX
"the Engineering Lead is thinking..." bubbles during Phase 2 would make the multi-agent process feel more alive and give users a sense of progress rather than a blank wait. The routing phase already shows status text; specialists don't yet.
08
Retry per specialist UX
API errors in Phase 2 are swallowed and displayed as [Name unavailable]. A per-specialist retry button on the bubble, or a global retry for the last query, would significantly improve experience when a provider has a transient failure.
09
Modular codebase Maintainability
The app is ~4,300 lines in a single HTML file. Introducing a minimal build step (Vite + vanilla JS) would enable code splitting, proper imports, and better separation of concerns — without losing the zero-backend philosophy.
10
Multi-tab / multi-device sync Reliability
Settings changed in one tab aren't reflected in other open sessions without a reload. A Supabase Realtime subscription on the relevant tables, or a BroadcastChannel for same-device, would keep all sessions in sync.
11
Mobile-optimised layout Accessibility
The boardroom SVG and dual-panel layout are designed for desktop. A fully rethought mobile layout — collapsible team panel, bottom chat bar, card-based specialist responses — would open Startup Staff to on-the-go use.
Roadmap

Known Limitations

API keys in browser memory. window.providerConfigs holds all API keys in plaintext during the session. Any JS running on the page — including browser extensions — can read them. Mitigation: use a backend proxy (see roadmap #04) and always set usage limits at your provider's dashboard.
No rate limiting. Nothing prevents rapid-fire queries. With all six specialists configured, a single query triggers up to 9 API calls. This can result in significant API cost burn without control. Use provider-level rate limits and spending caps as a safeguard.
Context window growth. chatHistory is never pruned. Long conversations passed to the CoS in Phase 1 and Phase 3 will eventually exceed model context windows, causing API errors. Session summarisation partially mitigates this but doesn't truncate the raw history array.
URL KB fetch CORS. URL fetching fails for most sites due to browser CORS policies. Errors appear as [Fetch failed] in the KB content. Use uploaded files for reliable knowledge injection.
Silent DB write failures. Some Supabase upserts are fire-and-forget — errors are not surfaced to the user. In-memory state and database can diverge silently. If settings seem not to stick, try re-saving and refreshing.
AI responses are not professional advice. This is an AI-powered tool. Legal, financial, and medical outputs are starting points for your own research — not substitutes for qualified professionals.
